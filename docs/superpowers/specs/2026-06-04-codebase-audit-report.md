## Layer 1: Turtle Runtime

### protocol.lua

- [BUG] `proto.send` ŌĆö Hardcodes `proto.CH_SERVER` (1) as the reply channel for every transmit, regardless of the actual sender. This is correct for turtleŌåÆserver traffic but wrong for serverŌåÆturtle (CH_BROADCAST/CH_PRIVATE), warehouse (CH_WAREHOUSE), and turtleŌåöturtle (CH_LOCAL) messages: their `replyChannel` metadata falsely advertises channel 1. Currently latent because `proto.receive` discards the reply channel (`repCh` is captured but never used), so no code replies on it ŌĆö but any future code that relies on `replyChannel` to route a response would send it to the server instead of the real sender. Should pass the sender's own listen channel as the reply channel.
- [OVERSIGHT] `proto.send` ŌĆö Calls `textutils.serialise(msg)` on every send, while `proto.receive` accepts BOTH a raw table (`type(raw) == "table"`) and a serialised string. The send path never transmits a raw table, so the table branch in receive is dead unless some other (non-proto) sender transmits raw tables. Asymmetry is harmless but worth noting; serialise also throws on payloads containing functions/recursive tables, which would crash the sender rather than fail gracefully.
- [OVERSIGHT] `proto.receive` ŌĆö `textutils.unserialise(raw)` returns `nil` on malformed input; the code guards with `local ok = msg ~= nil` and skips, so a corrupt/garbage modem message is silently dropped with no log. Acceptable, but failures are invisible for debugging.
- [OPT] `proto.receive` timer event ŌĆö The destructure `local event, side, ch, repCh, raw = os.pullEvent()` reuses modem-message field names for the timer case, where the 2nd return is actually the timer ID. The check `event == "timer" and side == timer` is CORRECT (for a timer event `side` holds the timer ID), but the variable name `side` is misleading and invites future bugs. No functional issue. Naming it `p1`/`arg2` or handling timer with its own pull would be clearer.
- [OVERSIGHT] `proto.MSG` orphan-from-name scan ŌĆö Most types map cleanly to a sender/handler. Two stand out by name: `ITEM_REQUEST`/`ITEM_READY` (older warehouse handshake) appear superseded by the newer `DELIVERY_ARRIVED`/`CHESTS_READY`/`ITEMS_READY`/`BATCH_DONE`/`ITEMS_DONE` handshake added below ŌĆö possible dead/duplicate protocol pair to verify against actual senders/handlers. `TURTLE_QUERY`/`TURTLE_INFO` are documented as used by support turtles but the newer CH_LOCAL follow signals (`POSITION_UPDATE`, `ASCENDING`, `DESCENDED`) may make the server-mediated query path redundant. Flagged for cross-file confirmation in later tasks (handler/sender presence not verified here).
- No issue with the timer-ID check (Check 1), `proto.decode` MSG lookup (Check 4 ŌĆö every `proto.MSG` entry has key == value, so `proto.MSG[msg.type]` validation succeeds for all valid types), terminate handling (Check 5 ŌĆö `proto.receive` uses `os.pullEvent()`, the filtering form that re-throws `terminate` as an error, so Ctrl-T shutdown is NOT swallowed; `os.pullEventRaw` would have been the bug), or channel separation (Check 7 ŌĆö channel is an explicit caller argument to `proto.send`, so CH_LOCAL messages are not auto-routed to CH_SERVER; only the reply-channel metadata is wrong, per the [BUG] above).

### waypoints.lua

- [OVERSIGHT] Hole coordinate literals duplicated across definitions ŌĆö The X/Z literals `143`/`-2813` appear in `DISPATCH_HOLE`, `DISPATCH_STAGING` (x only), and `WORLD_EXIT`; `228`/`-2782` appear in both `ARRIVALS_HOLE` and `WORLD_ENTRY`. `WORLD_EXIT`/`WORLD_ENTRY` are by definition "below the dispatch/arrivals hole," yet they restate the raw coords instead of referencing `W.DISPATCH_HOLE.x/.z` and `W.ARRIVALS_HOLE.x/.z`. A single layout move of either hole requires editing the coordinate in two-to-three places, and forgetting one would put the underground entry/exit out of vertical alignment with its hole (turtle ascends/descends into a wall). Derive `WORLD_EXIT`/`WORLD_ENTRY` from the hole tables (only Y differs).
- [OVERSIGHT] Route builders deref `dock` fields with no internal nil guard ŌĆö `internalReturnRouteFrom`, `returnAisleZ`, `dockFacing`, `departureRoute`, `supportDepartureRoute`, `returnRoute`, and `aisleZ` all access `dock.z` / `dock.x` / `dock.junction.x` without checking `dock` (or its fields) for nil inside the helper itself. All current callers are in `turtle_base.lua` and each guards at the call site with `if not _self.dock then return true end`, so no unguarded call path exists today ŌĆö this is latent. Any future caller added without that guard would hit "attempt to index a nil value" with no clear error message pointing back to the missing dock assignment.

Checks that passed (no issue): single-axis waypoint rule in `internalReturnRouteFrom` (all 5 steps change exactly one of X/Z relative to the prior step, starting from `fromX,fromZ`); the `internalReturnRoute` compatibility wrapper (calls `internalReturnRouteFrom(dock, dock.junction.x, W.RED_Z)` ŌĆö argument order matches the spec; its only quirk is a harmless duplicate first waypoint when `fromX==junction.x` and `fromZ==RED_Z`); `returnAisleZ` does not index by `dock.row` at all (it derives from `dock.z`), so an out-of-range/nil `row` is irrelevant to it; `dockFacing` returns only `2` or `0`, both valid {0,1,2,3}; and `CFG.FLOOR_Y` is referenced via the config constant everywhere inside this file (the raw `67` literal lives in `turtle_base.lua:622`, out of scope for this task).

### turtle_base.lua

The bridge push and its `resp.readAll()`/`resp.close()` ordering live in `server.js`/`central_server.lua` (server side); the receive-with-timeout primitive lives in `protocol.lua` (`proto.receive`). - [OVERSIGHT] Internal GPS resync ignores failure silently ŌĆö `applyMove` (line 172-174) calls `gpsSync()` every `GPS_RESYNC_INTERVAL` (50) moves but discards the return value. When `gps.locate()` returns nil (no satellite signal), `gpsSync` returns false and leaves `_self.pos` untouched (line 84-88), so the turtle keeps dead-reckoning with **no warning logged**. The same silent-fail pattern repeats at every bare `gpsSync()` call: `depart` (432), `returnToDock` (557, 575), `returnToDockInternal` (602, 618), and `base.init` (1063). The public `base.gpsSync` (line 92) does log a warning, but none of the internal callers use it. Not a crash (good ŌĆö no nil index), but drift accumulates invisibly. Recommend logging a WARN on internal resync failure, at least on the precision-critical resyncs before bay navigation.

- [OVERSIGHT] `bypassForward` floor gate uses raw literal `66` instead of `CFG.FLOOR_Y` ŌĆö Line 303 `if _self.pos.y < 66 then` guards the vertical (dig) bypass so the turtle never digs inside the surface depot. But the actual depot floor is `CFG.FLOOR_Y = 67` (waypoints.lua:11). A turtle standing on the floor is at y=67, so `67 < 66` is false ŌåÆ vertical bypass correctly skipped at the floor. However the threshold is off-by-one in intent: the comment says "Inside the surface depot (y >= 66) never dig", yet only y>=66 blocks it, meaning a turtle at **y=66** (one block below floor, e.g. mid-descent into a hole) is also treated as "depot" and refused a vertical bypass, while everything strictly below 66 (underground travel at y=60) is allowed. The threshold happens to work for the two real cases (floor=67 blocked, underground=60 allowed) but it is a hardcoded magic number that should be `_self.pos.y < W.DISPATCH_HOLE.y` or `< CFG.FLOOR_Y`. See also the duplicate-literal finding below.

- [OVERSIGHT] Hardcoded floor-Y literals `66`/`67` scattered instead of `CFG.FLOOR_Y` ŌĆö Raw Y constants appear at: line 303 (`66`, bypass gate), 515 (`local FLOOR_Y = 67`, returnToDock), 622 (`move.to(..., 67, ...)`, returnToDockInternal drift correction), 1065 (`p.y >= 66`, init inside-building test), and 1133 (`_self.pos.y >= 67`, RECALL inside-building test). `waypoints.lua` exports `W.FLOOR_Y`/`CFG.FLOOR_Y = 67`, and `returnToDock` even defines its own local `FLOOR_Y = 67` rather than importing it. The init test uses 66 while the RECALL test uses 67 for the *same* "inside building" predicate (lines 1065 vs 1133) ŌĆö an inconsistency: a turtle at exactly y=66 is "inside" for init homing but "outside" for RECALL, so RECALL would route it via the full underground `returnToDock` instead of the internal taxiway. Any floor-height change requires editing five places and they currently disagree.

- [OVERSIGHT] Building-bounds box duplicated and divergent ŌĆö The "inside building" bounding box `x in [143,228], z in [-2817,-2782]` is hardcoded twice (init lines 1066-1067, RECALL lines 1134-1135) with the only difference being the Y test (66 vs 67, above). These magic coordinates duplicate `W.DISPATCH_HOLE`/`W.ARRIVALS_HOLE` extents from waypoints and would silently break if the depot footprint moves. Extract a single `W.isInsideBuilding(pos)` helper.

- [BUG] `returnToDockInternal` route failure leaves status stuck at RETURNING ŌĆö Lines 614-616: if `move.followRoute(route)` fails partway (e.g. permanently blocked aisle), the function returns `false, "internal return failed: ..."` having already set `proto.STATUS.RETURNING` at line 609. Nothing resets the status or notifies the server, so the turtle reports RETURNING indefinitely while physically stuck mid-aisle. Same shape in `returnToDock` (lines 531, 572) and `depart` (443, 482) ŌĆö all return false on route failure with status left mid-transition and no `sendFailed`/recovery. The caller (delivery/support job handler) must handle the false return; if it doesn't, the turtle is wedged. Flagged for cross-check against the role handlers.

- [BUG] No fuel reserve sized for the return trip ŌĆö `fuel.isCritical()` triggers at a flat `FUEL_CRITICAL = 200` (line 14, 645). There is no calculation reserving enough fuel to reach the dock from the current position; 200 is a fixed floor regardless of how far away the turtle is. If a job takes the turtle far enough that it drops below 200 only after passing the point of no return, `ensureFuel` (859) will deploy the entangled chest mid-journey to recover ŌĆö which works *only* if the chest has coal. If the chest is empty, `ensureFuel` enters an unbounded `while fuel.isCritical()` loop (line 860) that blocks the turtle forever (sending ERROR heartbeats every 5s, 6 iterations = 30s per outer retry cycle), never aborting the job or returning. Mid-job fuel exhaustion is therefore a soft-hang, not a clean failure. There is no give-up / escalate-and-park path, so the delivery pair is locked until a human refills the chest.

- [BUG] `ensureFuel` infinite block on empty chest ŌĆö (detail of the above) Lines 860-881: the outer `while fuel.isCritical()` only exits when fuel is no longer critical. If `refuelFromChest` returns false (empty/unreachable chest), the inner for-loop runs 6 ├Ś `sleep(5)` (sending a heartbeat each iteration, one every 5s, 30s total per cycle) then re-loops the outer `while` forever. There is no max-retry / escalate-and-park path. Turtle stays "alive" to the server (heartbeats with STATUS.ERROR) but never progresses, permanently locking the delivery pair until human intervention.

- [OVERSIGHT] `requestItems` uses the legacy `ITEM_REQUEST`/`ITEM_READY` pair ŌĆö Lines 1001-1012 still send `proto.MSG.ITEM_REQUEST` and await `proto.MSG.ITEM_READY`, the older warehouse handshake flagged as possibly superseded in the protocol.lua audit. Confirm whether any current job handler calls `base.requestItems`; if the live path uses the newer `DELIVERY_ARRIVED`/`CHESTS_READY`/`ITEMS_READY` handshake, this is dead code. [PROTOCOL] pending sender/handler confirmation.

- [OVERSIGHT] `_self.facing` persists across jobs and is never re-validated by GPS ŌĆö Module-level `_self` (lines 24-39) holds `pos`, `facing`, `status`, `jobId`, `partnerId`, `busy`, `recalled`, `moveCount` for the whole process lifetime. `gpsSync` corrects `pos` but **never re-derives `facing`** (only the boot-time `detectFacing` does, line 104). If a chunk-unload/reload or a physical bump rotates the turtle without going through `move.turnLeft/Right`, `facing` silently desyncs from reality and every subsequent `applyMove` mistracks position until the next 50-move GPS resync masks it (position only, not heading). `move.to`'s per-step facing recalc partially self-heals X/Z drift, but a wrong heading still produces one wrong move per axis before correction. Re-running `detectFacing` after a GPS resync, or on job start, would harden this.

### delivery_turtle.lua

- [BUG] **`os.clock()` used for every deadline instead of `os.epoch("utc")` (Check 1 + 7).** Every timeout in the file is built on `os.clock()`, which returns CPU time and runs slow under parallel-coroutine load (the turtle runs `controlLoop` + `jobRunner` via `parallel.waitForAny`, plus this file's own nested wait loops). Affected sites: `waitForAny` (lines 25, 26, 29, 32), the queue/CHESTS_READY wait loop (lines 328 `deadline`, 329 `lastIReq`, 331, 333, 348, 351, 356). Real consequence: the "10 min" queue deadline (`os.clock() + 600`, line 328) and the 60s ITEM_REQUEST re-send interval (line 351) both expire *later* than intended in wall-clock terms, and the printed "%ds remaining" (line 347-348) understates how long the turtle has actually been waiting. Inconsistent with the warehouse side, which times out on wall-clock (`arrivalTimeout`/`msgTimeout`), so the turtle can still be "waiting" by its clock long after the warehouse has aborted the job. Should use `os.epoch("utc")/1000` (or `os.epoch("local")`) for all deadline math.

- [BUG] **First-phase (WAIT_ARRIVE) wait loop never breaks on RECALL (Check 3).** The queue/CHESTS_READY loop (lines 331-372) calls `waitForAny(...)`, and `waitForAny` *does* return `nil` immediately when `base.isRecalled()` (line 27). But the enclosing `while os.clock() < deadline` (line 331) has **no `isRecalled()` check** ŌĆö when `waitForAny` returns nil the loop treats it as a normal "no message" tick: it re-sends DELIVERY_ARRIVED, increments `waitTick`, possibly re-sends ITEM_REQUEST, and loops again. So a RECALL received while the turtle is waiting for CHESTS_READY does **not** abort here; the turtle keeps pinging the warehouse for up to the full 600s deadline (or until `warehouse_chest_timeout`), spamming DELIVERY_ARRIVED the entire time. Meanwhile the control loop already fired `base.sendFailed("recalled")` (turtle_base.lua:1129) which cleared `partnerId`, status, and `busy` and told the server the job failed ŌĆö so the turtle is reporting IDLE/failed to the server while its job coroutine is still actively looping and transmitting. It only escapes when the deadline elapses, then hits `warehouse_chest_timeout` and finally digs up the EC. Contrast the *second* phase loop (lines 431-472) which **does** check `base.isRecalled()` (line 438) at the top and breaks cleanly. Add the same `if base.isRecalled() then break end` to the first loop. (Note this is the loop the v1.3.35 "send DELIVERY_ARRIVED when no message" patch touched, so the regression is recent.)

- [OVERSIGHT] **`params.destination` / x,y,z nil check is absent (Check 6).** `local d = params.destination` (line 211) is dereferenced as `d.x`, `d.z` in the first `sendProgress` (line 259) and then `base.move.to(d.x, UNDERGROUND_Y, d.z)` (line 261). If `params.destination` is nil, line 259's `string.format(... d.x ...)` throws "attempt to index a nil value (local 'd')" before move is even reached. If `d` exists but `d.x`/`d.z` is nil, `move.to` compares `_self.pos.x ~= nil` ŌåÆ the `while _self.pos.x ~= tx` loop in turtle_base never terminates correctly / `_self.pos.y < ty` throws on the nil Y. Either way the job handler crashes; `jobRunner`'s `pcall` (turtle_base.lua:1178) catches it and calls `sendFailed(tostring(err))`, so it's a caught crash rather than a hang ŌĆö but the failure reason is an opaque Lua error string, and the EC was never placed so no cleanup is needed. Recommend an explicit guard at job start: `if not d or not d.x or not d.y or not d.z then return base.sendFailed("missing_destination", false) end`.

- [OVERSIGHT] **Chest placement has no slot-occupied / wrong-item / inventory-full safety (Check 5).** (a) `placeChestAt` ŌåÆ `placeDownClear` (lines 40-43) digs whatever is below before placing, so a world block underneath is handled ŌĆö but `turtle.placeDown()`'s return value is **discarded** at line 49 and 293; if placement fails (e.g. an unbreakable block, or an entity occupying the space) the code proceeds as though the chest is down and later `suckDown`/`dropDown` silently do nothing, stalling the handshake. (b) Inventory-full during the pull: `suckDown(1)` is called `chestCount` times (lines 384-386) but its return is only counted, never acted on ŌĆö if slots 1-15 are unexpectedly full the pulled count is lower than requested and placement silently does fewer chests; the warehouse is told the real `#chestPositions` (line 419-421) so it's consistent, but the under-delivery is invisible. (c) **Wrong item type from the entangled chest:** the placement loop (lines 394-401) scans for `it.name:find("chest")`, so a non-chest item pulled out of the EC is simply skipped and left in inventory ŌĆö it never gets placed and never gets dumped (the road-debris dump already happened at line 299-308, before the pull). That stray item then rides home in a non-protected slot and, on the *next* job, will be dropped into the next warehouse's EC as "debris." Low severity but a slow contamination path.

- [OVERSIGHT] **`turtle.placeDown()` return ignored for the entangled chest too (Check 5/10-adjacent).** Line 292-293 selects EC_SLOT and calls `placeDownClear()` without checking success. If the EC fails to place (block below couldn't be dug ŌĆö bedrock, protected region, another turtle), the subsequent `dropDown` debris-dump (line 304) and every `suckDown` operate against air/no inventory, and the warehouse handshake stalls until the 600s queue timeout. CHECKPOINT 2 (PRE-PLACE, line 287) only verifies the chest is *in slot 16 before* placing, not that it actually went down.

- [OPT] **`lastIReq` is never reset once the first phase ends, but that is harmless (Check 8).** `lastIReq` (line 329) governs the 60s ITEM_REQUEST re-send and lives only inside the first wait loop (lines 331-372). The second phase (batch loop) re-sends `CHESTS_PLACED`, not `ITEM_REQUEST`, and never reads `lastIReq` again, so there is no stale-timer misfire across batches ŌĆö the concern in the task description does not materialize because `lastIReq`'s scope is confined to the queue phase. No change needed; noted for completeness. (If the first loop is ever refactored to re-enter after a batch, reset `lastIReq` then.)

- [BUG] **`chestIdx` is never reset between batches, so later batches can't reuse earlier chests (Check, distribution logic).** `chestIdx` (line 429) starts at 1 and only ever *increments* (line 464) when a chest reports full. Across multiple ITEMS_READY batches, once `chestIdx` advances past a chest it never returns to it ŌĆö even if a later batch is small and the current chest still has room, the turtle keeps filling from `chestIdx` forward. More seriously, when `chestIdx > #chestPositions` the inner `while chestIdx <= #chestPositions` (line 458) never executes, so a batch pulled into the turtle (line 453-455) is sucked up but **never dropped into any chest** ŌĆö those items stay in turtle inventory, `BATCH_DONE` is still sent (line 470), and the items ride home undelivered. This is the failure mode when total item volume exceeds `#chestPositions` chests' capacity. At minimum the overflow case should be detected and reported (sendFailed / sendProgress), not silently swallowed.

- [PROTOCOL/OK] **JOB_ABORT to support on abort is handled by `base.sendFailed` (Check 4) ŌĆö OK.** Every abort exit in this file goes through `base.sendFailed(...)` (lines 226, 253, 263, 277, 287's failure, 362, 376, 490) or the in-loop JOB_ABORT branch. `base.sendFailed` (turtle_base.lua:986-988) signals `JOB_ABORT` to `_self.partnerId` on CH_LOCAL before clearing it, and `support_turtle.lua:143-145` handles JOB_ABORT mid-job by returning to dock. On RECALL, the control loop calls `sendFailed("recalled")` itself (turtle_base.lua:1129), so the support turtle is released even though the delivery job coroutine is still (buggily, per the Check 3 finding) looping. So the support turtle is **not** orphaned on any abort path. No fix needed for Check 4 itself.

- [OK] **RETURN_TO_DOCK to support on normal completion (Check 9) ŌĆö OK with one caveat.** Delivery clears `partnerId` at ascent (line 269) and restores it before descent (lines 506-507, guarded by `not isRecalled()`), so `base.returnToDock` captures a non-nil `savedPartnerId` and emits `RETURN_TO_DOCK` on CH_LOCAL after clearing the arrivals hole (turtle_base.lua:561-566). `support_turtle.lua:135` handles it. Caveat: on the RECALL/abort paths the descent restore (line 506) is skipped, so `partnerId` stays nil and `returnToDock` sends **no** RETURN_TO_DOCK ŌĆö but that's correct here because the support turtle already received JOB_ABORT and is returning on its own. No orphan.

- [OVERSIGHT] **EC pickup on abort is best-effort and unverified (Check, cleanup).** The two first-phase abort exits (line 361 JOB_ABORT, line 375 timeout) do `turtle.select(EC_SLOT); turtle.digDown()` then immediately `sendFailed`, with **no `checkEC` verification** that the delivery ender chest actually came back (unlike the clean path, which has CHECKPOINT 3 PRE-RETURN at line 484 with a retry). If the turtle wasn't perfectly centred over the placed EC, `digDown` misses, the EC is left in the world, and the turtle returns to dock without its delivery chest ŌĆö silently. Recommend routing these aborts through the same `checkEC("ABORT", ...)` recovery used on the clean path before returning.

- [OPT] **No `peripheral.wrap()` calls in this file (Check 10).** Delivery_turtle does all I/O via `turtle.*` builtins and `base.*` helpers; there are no `peripheral.wrap()` calls here, so the "wrap returns nil if not attached" hazard does not apply to this file. (Any peripheral wrapping happens in `turtle_base`/`protocol`, audited separately.) The modem itself is wrapped in `base.init`; if absent, `base.signalPartner*`/`sendToServer` no-op via the `if not _self.modem` guards rather than crashing.

- [OK] **Warehouse-side senders for all consumed messages exist (Check 2, cross-checked against warehouse.lua).** `CHESTS_READY` (warehouse.lua:317,332), `WAREHOUSE_QUEUED` (205,223,250,301), `ITEMS_READY` (378), `ITEMS_DONE` (367), and the `JOB_ABORT` abort path (`abortCurrent`, 359) all have real senders, and the turtle's emitted `DELIVERY_ARRIVED` / `CHESTS_PLACED` / `BATCH_DONE` / `ITEM_COLLECTED` / `ITEM_REQUEST` are all consumed there. The one asymmetry: the turtle ends the clean path with `ITEM_COLLECTED` (line 495), which the warehouse's `WAIT_DONE` state expects ŌĆö names line up. No orphaned message in the delivery handshake. **FSM timeout summary:** WAIT_ARRIVE loop has a 600s deadline ŌåÆ `warehouse_chest_timeout` (good, but see the RECALL gap above); the batch loop (Phase 2) has **no overall deadline at all** ŌĆö it loops `while true` with a 10s `waitForAny` and only exits on ITEMS_DONE, JOB_ABORT, or `isRecalled()`. If the warehouse goes silent mid-delivery (never sends ITEMS_DONE and never aborts), the turtle re-pings CHESTS_PLACED forever (line 443) with no escape ŌĆö a stall-without-timeout [BUG]: Phase 2 should carry its own wall-clock deadline mirroring the warehouse's `msgTimeout`.
