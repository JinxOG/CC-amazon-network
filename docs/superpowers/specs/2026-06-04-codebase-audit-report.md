## Layer 1: Turtle Runtime

### protocol.lua

- [BUG] `proto.send` ‚Äî Hardcodes `proto.CH_SERVER` (1) as the reply channel for every transmit, regardless of the actual sender. This is correct for turtle‚Üíserver traffic but wrong for server‚Üíturtle (CH_BROADCAST/CH_PRIVATE), warehouse (CH_WAREHOUSE), and turtle‚Üîturtle (CH_LOCAL) messages: their `replyChannel` metadata falsely advertises channel 1. Currently latent because `proto.receive` discards the reply channel (`repCh` is captured but never used), so no code replies on it ‚Äî but any future code that relies on `replyChannel` to route a response would send it to the server instead of the real sender. Should pass the sender's own listen channel as the reply channel.
- [OVERSIGHT] `proto.send` ‚Äî Calls `textutils.serialise(msg)` on every send, while `proto.receive` accepts BOTH a raw table (`type(raw) == "table"`) and a serialised string. The send path never transmits a raw table, so the table branch in receive is dead unless some other (non-proto) sender transmits raw tables. Asymmetry is harmless but worth noting; serialise also throws on payloads containing functions/recursive tables, which would crash the sender rather than fail gracefully.
- [OVERSIGHT] `proto.receive` ‚Äî `textutils.unserialise(raw)` returns `nil` on malformed input; the code guards with `local ok = msg ~= nil` and skips, so a corrupt/garbage modem message is silently dropped with no log. Acceptable, but failures are invisible for debugging.
- [OPT] `proto.receive` timer event ‚Äî The destructure `local event, side, ch, repCh, raw = os.pullEvent()` reuses modem-message field names for the timer case, where the 2nd return is actually the timer ID. The check `event == "timer" and side == timer` is CORRECT (for a timer event `side` holds the timer ID), but the variable name `side` is misleading and invites future bugs. No functional issue. Naming it `p1`/`arg2` or handling timer with its own pull would be clearer.
- [OVERSIGHT] `proto.MSG` orphan-from-name scan ‚Äî Most types map cleanly to a sender/handler. Two stand out by name: `ITEM_REQUEST`/`ITEM_READY` (older warehouse handshake) appear superseded by the newer `DELIVERY_ARRIVED`/`CHESTS_READY`/`ITEMS_READY`/`BATCH_DONE`/`ITEMS_DONE` handshake added below ‚Äî possible dead/duplicate protocol pair to verify against actual senders/handlers. `TURTLE_QUERY`/`TURTLE_INFO` are documented as used by support turtles but the newer CH_LOCAL follow signals (`POSITION_UPDATE`, `ASCENDING`, `DESCENDED`) may make the server-mediated query path redundant. Flagged for cross-file confirmation in later tasks (handler/sender presence not verified here).
- No issue with the timer-ID check (Check 1), `proto.decode` MSG lookup (Check 4 ‚Äî every `proto.MSG` entry has key == value, so `proto.MSG[msg.type]` validation succeeds for all valid types), terminate handling (Check 5 ‚Äî `proto.receive` uses `os.pullEvent()`, the filtering form that re-throws `terminate` as an error, so Ctrl-T shutdown is NOT swallowed; `os.pullEventRaw` would have been the bug), or channel separation (Check 7 ‚Äî channel is an explicit caller argument to `proto.send`, so CH_LOCAL messages are not auto-routed to CH_SERVER; only the reply-channel metadata is wrong, per the [BUG] above).

### waypoints.lua

- [OVERSIGHT] Hole coordinate literals duplicated across definitions ‚Äî The X/Z literals `143`/`-2813` appear in `DISPATCH_HOLE`, `DISPATCH_STAGING` (x only), and `WORLD_EXIT`; `228`/`-2782` appear in both `ARRIVALS_HOLE` and `WORLD_ENTRY`. `WORLD_EXIT`/`WORLD_ENTRY` are by definition "below the dispatch/arrivals hole," yet they restate the raw coords instead of referencing `W.DISPATCH_HOLE.x/.z` and `W.ARRIVALS_HOLE.x/.z`. A single layout move of either hole requires editing the coordinate in two-to-three places, and forgetting one would put the underground entry/exit out of vertical alignment with its hole (turtle ascends/descends into a wall). Derive `WORLD_EXIT`/`WORLD_ENTRY` from the hole tables (only Y differs).
- [OVERSIGHT] Route builders deref `dock` fields with no internal nil guard ‚Äî `internalReturnRouteFrom`, `returnAisleZ`, `dockFacing`, `departureRoute`, `supportDepartureRoute`, `returnRoute`, and `aisleZ` all access `dock.z` / `dock.x` / `dock.junction.x` without checking `dock` (or its fields) for nil inside the helper itself. All current callers are in `turtle_base.lua` and each guards at the call site with `if not _self.dock then return true end`, so no unguarded call path exists today ‚Äî this is latent. Any future caller added without that guard would hit "attempt to index a nil value" with no clear error message pointing back to the missing dock assignment.

Checks that passed (no issue): single-axis waypoint rule in `internalReturnRouteFrom` (all 5 steps change exactly one of X/Z relative to the prior step, starting from `fromX,fromZ`); the `internalReturnRoute` compatibility wrapper (calls `internalReturnRouteFrom(dock, dock.junction.x, W.RED_Z)` ‚Äî argument order matches the spec; its only quirk is a harmless duplicate first waypoint when `fromX==junction.x` and `fromZ==RED_Z`); `returnAisleZ` does not index by `dock.row` at all (it derives from `dock.z`), so an out-of-range/nil `row` is irrelevant to it; `dockFacing` returns only `2` or `0`, both valid {0,1,2,3}; and `CFG.FLOOR_Y` is referenced via the config constant everywhere inside this file (the raw `67` literal lives in `turtle_base.lua:622`, out of scope for this task).

- [OVERSIGHT] Internal GPS resync ignores failure silently ‚Äî `applyMove` (line 172-174) calls `gpsSync()` every `GPS_RESYNC_INTERVAL` (50) moves but discards the return value. When `gps.locate()` returns nil (no satellite signal), `gpsSync` returns false and leaves `_self.pos` untouched (line 84-88), so the turtle keeps dead-reckoning with **no warning logged**. - [OVERSIGHT] `bypassForward` floor gate uses raw literal `66` instead of `CFG.FLOOR_Y` ‚Äî Line 303 `if _self.pos.y < 66 then` guards the vertical (dig) bypass so the turtle never digs inside the surface depot. But the actual depot floor is `CFG.FLOOR_Y = 67` (waypoints.lua:11). A turtle standing on the floor is at y=67, so `67 < 66` is false ‚Üí vertical bypass correctly skipped at the floor. However the threshold is off-by-one in intent: the comment says "Inside the surface depot (y >= 66) never dig", yet only y>=66 blocks it, meaning a turtle at **y=66** (one block below floor, e.g. mid-descent into a hole) is also treated as "depot" and refused a vertical bypass, while everything strictly below 66 (underground travel at y=60) is allowed. The threshold happens to work for the two real cases (floor=67 blocked, underground=60 allowed) but it is a hardcoded magic number that should be `_self.pos.y < W.DISPATCH_HOLE.y` or `< CFG.FLOOR_Y`. See also the duplicate-literal finding below.

- [OVERSIGHT] Hardcoded floor-Y literals `66`/`67` scattered instead of `CFG.FLOOR_Y` ‚Äî Raw Y constants appear at: line 303 (`66`, bypass gate), 515 (`local FLOOR_Y = 67`, returnToDock), 622 (`move.to(..., 67, ...)`, returnToDockInternal drift correction), 1065 (`p.y >= 66`, init inside-building test), and 1133 (`_self.pos.y >= 67`, RECALL inside-building test). `waypoints.lua` exports `W.FLOOR_Y`/`CFG.FLOOR_Y = 67`, and `returnToDock` even defines its own local `FLOOR_Y = 67` rather than importing it. - [BUG] No fuel reserve sized for the return trip ‚Äî `fuel.isCritical()` triggers at a flat `FUEL_CRITICAL = 200` (line 14, 645). There is no calculation reserving enough fuel to reach the dock from the current position; 200 is a fixed floor regardless of how far away the turtle is. If a job takes the turtle far enough that it drops below 200 only after passing the point of no return, `ensureFuel` (859) will deploy the entangled chest mid-journey to recover ‚Äî which works *only* if the chest has coal. If the chest is empty, `ensureFuel` enters an unbounded `while fuel.isCritical()` loop (line 860) that blocks the turtle forever (sending ERROR heartbeats every 5s, 6 iterations = 30s per outer retry cycle), never aborting the job or returning. Mid-job fuel exhaustion is therefore a soft-hang, not a clean failure. There is no give-up / escalate-and-park path, so the delivery pair is locked until a human refills the chest.

- [BUG] `ensureFuel` infinite block on empty chest ‚Äî (detail of the above) Lines 860-881: the outer `while fuel.isCritical()` only exits when fuel is no longer critical. If `refuelFromChest` returns false (empty/unreachable chest), the inner for-loop runs 6 √ó `sleep(5)` (sending a heartbeat each iteration, one every 5s, 30s total per cycle) then re-loops the outer `while` forever. There is no max-retry / escalate-and-park path. Turtle stays "alive" to the server (heartbeats with STATUS.ERROR) but never progresses, permanently locking the delivery pair until human intervention.

### delivery_turtle.lua

- [BUG] **`os.clock()` used for every deadline instead of `os.epoch("utc")` (Check 1 + 7).** Every timeout in the file is built on `os.clock()`, which returns CPU time and runs slow under parallel-coroutine load (the turtle runs `controlLoop` + `jobRunner` via `parallel.waitForAny`, plus this file's own nested wait loops). Affected sites: `waitForAny` (lines 25, 26, 29, 32), the queue/CHESTS_READY wait loop (lines 328 `deadline`, 329 `lastIReq`, 331, 333, 348, 351, 356). Real consequence: the "10 min" queue deadline (`os.clock() + 600`, line 328) and the 60s ITEM_REQUEST re-send interval (line 351) both expire *later* than intended in wall-clock terms, and the printed "%ds remaining" (line 347-348) understates how long the turtle has actually been waiting. Inconsistent with the warehouse side, which times out on wall-clock (`arrivalTimeout`/`msgTimeout`), so the turtle can still be "waiting" by its clock long after the warehouse has aborted the job. Should use `os.epoch("utc")/1000` (or `os.epoch("local")`) for all deadline math.

- [BUG] **First-phase (WAIT_ARRIVE) wait loop never breaks on RECALL (Check 3).** The queue/CHESTS_READY loop (lines 331-372) calls `waitForAny(...)`, and `waitForAny` *does* return `nil` immediately when `base.isRecalled()` (line 27). But the enclosing `while os.clock() < deadline` (line 331) has **no `isRecalled()` check** ‚Äî when `waitForAny` returns nil the loop treats it as a normal "no message" tick: it re-sends DELIVERY_ARRIVED, increments `waitTick`, possibly re-sends ITEM_REQUEST, and loops again. So a RECALL received while the turtle is waiting for CHESTS_READY does **not** abort here; the turtle keeps pinging the warehouse for up to the full 600s deadline (or until `warehouse_chest_timeout`), spamming DELIVERY_ARRIVED the entire time. Meanwhile the control loop already fired `base.sendFailed("recalled")` (turtle_base.lua:1129) which cleared `partnerId`, status, and `busy` and told the server the job failed ‚Äî so the turtle is reporting IDLE/failed to the server while its job coroutine is still actively looping and transmitting. It only escapes when the deadline elapses, then hits `warehouse_chest_timeout` and finally digs up the EC. Contrast the *second* phase loop (lines 431-472) which **does** check `base.isRecalled()` (line 438) at the top and breaks cleanly. Add the same `if base.isRecalled() then break end` to the first loop. (Note this is the loop the v1.3.35 "send DELIVERY_ARRIVED when no message" patch touched, so the regression is recent.)

- [OVERSIGHT] **`params.destination` / x,y,z nil check is absent (Check 6).** `local d = params.destination` (line 211) is dereferenced as `d.x`, `d.z` in the first `sendProgress` (line 259) and then `base.move.to(d.x, UNDERGROUND_Y, d.z)` (line 261). If `params.destination` is nil, line 259's `string.format(... d.x ...)` throws "attempt to index a nil value (local 'd')" before move is even reached. If `d` exists but `d.x`/`d.z` is nil, `move.to` compares `_self.pos.x ~= nil` ‚Üí the `while _self.pos.x ~= tx` loop in turtle_base never terminates correctly / `_self.pos.y < ty` throws on the nil Y. Either way the job handler crashes; `jobRunner`'s `pcall` (turtle_base.lua:1178) catches it and calls `sendFailed(tostring(err))`, so it's a caught crash rather than a hang ‚Äî but the failure reason is an opaque Lua error string, and the EC was never placed so no cleanup is needed. Recommend an explicit guard at job start: `if not d or not d.x or not d.y or not d.z then return base.sendFailed("missing_destination", false) end`.

- [OVERSIGHT] **Chest placement has no slot-occupied / wrong-item / inventory-full safety (Check 5).** (a) `placeChestAt` ‚Üí `placeDownClear` (lines 40-43) digs whatever is below before placing, so a world block underneath is handled ‚Äî but `turtle.placeDown()`'s return value is **discarded** at line 49 and 293; if placement fails (e.g. an unbreakable block, or an entity occupying the space) the code proceeds as though the chest is down and later `suckDown`/`dropDown` silently do nothing, stalling the handshake. (b) Inventory-full during the pull: `suckDown(1)` is called `chestCount` times (lines 384-386) but its return is only counted, never acted on ‚Äî if slots 1-15 are unexpectedly full the pulled count is lower than requested and placement silently does fewer chests; the warehouse is told the real `#chestPositions` (line 419-421) so it's consistent, but the under-delivery is invisible. (c) **Wrong item type from the entangled chest:** the placement loop (lines 394-401) scans for `it.name:find("chest")`, so a non-chest item pulled out of the EC is simply skipped and left in inventory ‚Äî it never gets placed and never gets dumped (the road-debris dump already happened at line 299-308, before the pull). That stray item then rides home in a non-protected slot and, on the *next* job, will be dropped into the next warehouse's EC as "debris." Low severity but a slow contamination path.

- [OVERSIGHT] **`turtle.placeDown()` return ignored for the entangled chest too (Check 5/10-adjacent).** Line 292-293 selects EC_SLOT and calls `placeDownClear()` without checking success. If the EC fails to place (block below couldn't be dug ‚Äî bedrock, protected region, another turtle), the subsequent `dropDown` debris-dump (line 304) and every `suckDown` operate against air/no inventory, and the warehouse handshake stalls until the 600s queue timeout. CHECKPOINT 2 (PRE-PLACE, line 287) only verifies the chest is *in slot 16 before* placing, not that it actually went down.

- [OPT] **`lastIReq` is never reset once the first phase ends, but that is harmless (Check 8).** `lastIReq` (line 329) governs the 60s ITEM_REQUEST re-send and lives only inside the first wait loop (lines 331-372). The second phase (batch loop) re-sends `CHESTS_PLACED`, not `ITEM_REQUEST`, and never reads `lastIReq` again, so there is no stale-timer misfire across batches ‚Äî the concern in the task description does not materialize because `lastIReq`'s scope is confined to the queue phase. No change needed; noted for completeness. (If the first loop is ever refactored to re-enter after a batch, reset `lastIReq` then.)

- [BUG] **`chestIdx` is never reset between batches, so later batches can't reuse earlier chests (Check, distribution logic).** `chestIdx` (line 429) starts at 1 and only ever *increments* (line 464) when a chest reports full. Across multiple ITEMS_READY batches, once `chestIdx` advances past a chest it never returns to it ‚Äî even if a later batch is small and the current chest still has room, the turtle keeps filling from `chestIdx` forward. More seriously, when `chestIdx > #chestPositions` the inner `while chestIdx <= #chestPositions` (line 458) never executes, so a batch pulled into the turtle (line 453-455) is sucked up but **never dropped into any chest** ‚Äî those items stay in turtle inventory, `BATCH_DONE` is still sent (line 470), and the items ride home undelivered. This is the failure mode when total item volume exceeds `#chestPositions` chests' capacity. At minimum the overflow case should be detected and reported (sendFailed / sendProgress), not silently swallowed.

- [PROTOCOL/OK] **JOB_ABORT to support on abort is handled by `base.sendFailed` (Check 4) ‚Äî OK.** Every abort exit in this file goes through `base.sendFailed(...)` (lines 226, 253, 263, 277, 287's failure, 362, 376, 490) or the in-loop JOB_ABORT branch. `base.sendFailed` (turtle_base.lua:986-988) signals `JOB_ABORT` to `_self.partnerId` on CH_LOCAL before clearing it, and `support_turtle.lua:143-145` handles JOB_ABORT mid-job by returning to dock. On RECALL, the control loop calls `sendFailed("recalled")` itself (turtle_base.lua:1129), so the support turtle is released even though the delivery job coroutine is still (buggily, per the Check 3 finding) looping. So the support turtle is **not** orphaned on any abort path. No fix needed for Check 4 itself.

- [OK] **RETURN_TO_DOCK to support on normal completion (Check 9) ‚Äî OK with one caveat.** Delivery clears `partnerId` at ascent (line 269) and restores it before descent (lines 506-507, guarded by `not isRecalled()`), so `base.returnToDock` captures a non-nil `savedPartnerId` and emits `RETURN_TO_DOCK` on CH_LOCAL after clearing the arrivals hole (turtle_base.lua:561-566). `support_turtle.lua:135` handles it. Caveat: on the RECALL/abort paths the descent restore (line 506) is skipped, so `partnerId` stays nil and `returnToDock` sends **no** RETURN_TO_DOCK ‚Äî but that's correct here because the support turtle already received JOB_ABORT and is returning on its own. - [OVERSIGHT] **EC pickup on abort is best-effort and unverified (Check, cleanup).** The two first-phase abort exits (line 361 JOB_ABORT, line 375 timeout) do `turtle.select(EC_SLOT); turtle.digDown()` then immediately `sendFailed`, with **no `checkEC` verification** that the delivery ender chest actually came back (unlike the clean path, which has CHECKPOINT 3 PRE-RETURN at line 484 with a retry). - [OPT] **No `peripheral.wrap()` calls in this file (Check 10).** Delivery_turtle does all I/O via `turtle.*` builtins and `base.*` helpers; there are no `peripheral.wrap()` calls here, so the "wrap returns nil if not attached" hazard does not apply to this file. - [OK] **Warehouse-side senders for all consumed messages exist (Check 2, cross-checked against warehouse.lua).** `CHESTS_READY` (warehouse.lua:317,332), `WAREHOUSE_QUEUED` (205,223,250,301), `ITEMS_READY` (378), `ITEMS_DONE` (367), and the `JOB_ABORT` abort path (`abortCurrent`, 359) all have real senders, and the turtle's emitted `DELIVERY_ARRIVED` / `CHESTS_PLACED` / `BATCH_DONE` / `ITEM_COLLECTED` / `ITEM_REQUEST` are all consumed there. If the warehouse goes silent mid-delivery (never sends ITEMS_DONE and never aborts), the turtle re-pings CHESTS_PLACED forever (line 443) with no escape ‚Äî a stall-without-timeout [BUG]: Phase 2 should carry its own wall-clock deadline mirroring the warehouse's `msgTimeout`.

### support_turtle.lua

Scope note: support_turtle.lua is the thin job handler (155 lines); most of the actual CH_LOCAL movement logic (SUPPORT_STAGED handshake, ASCENDING/RETURN_TO_DOCK emission, POSITION_UPDATE broadcast) lives in `turtle_base.lua` `base.depart`/`base.returnToDock`/`applyMove`. Findings below cover both the handler and the base functions it relies on.

- [OVERSIGHT] **Movement blocks the message loop ‚Äî abort/hold signals are queued, not real-time (Check 1, 3, 7).** The follow loop (lines 97-149) handles a message, then on POSITION_UPDATE calls `base.move.to(prev.x, prev.y, prev.z)` (line 121), which is a **blocking multi-step navigation** that does not poll the modem. While support is mid-`move.to`, any `ASCENDING`, `RETURN_TO_DOCK`, or `JOB_ABORT` sent by delivery is buffered in the modem event queue and only acted on at the *next* `proto.receive` after the move completes. In practice delivery only moves one block between POSITION_UPDATEs so `move.to` is usually a single step, but if support has fallen behind (missed updates) a single `prev` can be several blocks away, lengthening the window where it ignores ASCENDING. Worse for the abort case: support keeps following toward `prev` for the full duration of that move before it reads JOB_ABORT ‚Äî it does not "stop mid-movement," it finishes the current move first. Not a crash, but the "hold immediately when delivery ascends" intent is not guaranteed.

- [BUG] **POSITION_UPDATE payload is not validated before use (Check 6).** Line 119-122 reads `local prev = msg.payload.prev` and guards only `if prev then`. It does not validate that `prev.x`, `prev.y`, `prev.z` are non-nil numbers. - [OK] **Queue advances correctly on completion; a concurrent DELIVERY_ARRIVED for `current` does not double-queue (Check 5).** On ITEM_COLLECTED (WAIT_DONE, line 403-406) the warehouse sets `current=nil`, clears batches, enters IDLE; the next `tick()` (fired by the 1s timer or the next message) pops `queue[1]` and starts serving it (lines 296-304). On the ITEMS_DONE‚ÜíWAIT_DONE path the same applies. Race check: a DELIVERY_ARRIVED arriving **while serving** `current` is caught by the auto-queue `known = current and current.turtleId == msg.from` guard (line 236), so it is not re-queued; it then falls through to `inboxPut` (line 285 via comment at 256) and is consumed as a benign re-ping in WAIT_PLACED (line 329) or ignored in other states (it sits in the inbox bucket and is only `inboxGet`-matched in WAIT_ARRIVE/WAIT_PLACED). No double-service. **One latent leak**: if a DELIVERY_ARRIVED for the *currently-served* turtle is buffered but never consumed (e.g. arrives during WAIT_BATCH/WAIT_DONE), it stays in `inbox[DELIVERY_ARRIVED]` forever ‚Äî the bucket is never swept on job completion, so stale entries accumulate across many jobs ([PERF]/[OVERSIGHT], slow unbounded inbox growth; the next job for the same turtleId could even mis-consume an old DELIVERY_ARRIVED in WAIT_ARRIVE and skip its real arrival wait).

- [OK] **`sendToServer` uses the correct channel and the server routes by jobId, not the `to` field (Check 6).** `sendToServer` (lines 52-55) transmits on `proto.CH_SERVER` (1), which is correct: the **server** listens on CH_SERVER (central_server.lua:828) and forwards warehouse replies to the turtle via `fwdToTurtle`, which routes by `payload.jobId ‚Üí job.assignedTo` (central_server.lua:612-617), NOT by the message `to` field. The warehouse receives server-forwarded messages on CH_WAREHOUSE (4), which it opens at line 46. So the channel pairing is correct (warehouse TX on CH_SERVER, RX on CH_WAREHOUSE; server TX on CH_WAREHOUSE, RX on CH_SERVER). [OPT] The `toId` argument passed to `sendToServer` (the turtle id) is effectively **ignored** by the server's routing (it routes solely on `jobId`); it's only cosmetic/log metadata. Every warehouse‚Üíturtle message therefore MUST include a valid `jobId` in the payload ‚Äî they all do (lines 206, 224, 251, 302, 318, 333, 367, 378, 388, 401). If a payload ever omitted jobId, the server would hit the NOT-FOUND‚ÜíJOB_ABORT path.

- [OK] **Turtle disconnect after CHESTS_READY is covered by the WAIT_PLACED timeout (Check 7).** After CHESTS_READY is sent (line 317), the warehouse enters WAIT_PLACED and waits for CHESTS_PLACED with a 120s `msgTimeout` (line 358). If the turtle disconnects/crashes before CHESTS_PLACED, the timeout fires ‚Üí `abortCurrent("CHESTS_PLACED timeout")` sweeps the EC (recovering the loaded chests back into RS) and returns to IDLE. It does **not** wait forever. Caveat (shared with Check 3): the abort is silent to the turtle/server ‚Äî no JOB_ABORT is sent from `abortCurrent`, so if the turtle reconnects after the abort it will re-ping DELIVERY_ARRIVED and be auto-queued fresh (correct recovery), but the loaded chests were already swept so it restarts cleanly.

- [BUG] **Item export does not verify stock atomically and silently under-delivers; a requested item missing from RS is only WARN-logged, never aborted (Check 8).** `checkStock` (lines 97-104) reads `rsBridge.getItem().amount` and logs a WARNING if `have < needed`, but **returns regardless** and the caller ignores the return (line 342 calls it purely for the side-effect log). `exportItem` (lines 92-95) then exports as many as RS has; if RS has fewer than requested, `moved < entry.count` and a "Short: m/n" line is logged (line 374-375) but the job **proceeds** ‚Äî the turtle is told ITEMS_READY and pulls whatever is in the EC, under-delivering silently to the destination. There is no abort, no re-queue, no message back to the server flagging the shortfall. A completely-missing item (`getItem` returns nil ‚Üí `have=0`) likewise only WARNs and continues with zero delivered for that item. Also: stock is checked at batch-build time (line 342, in WAIT_PLACED) but exported later batch-by-batch in SEND_BATCH ‚Äî between check and export, another consumer could drain RS, so even a passing `checkStock` doesn't guarantee the export. Recommend: on a hard shortfall, either abort the job with a distinct reason the server can surface, or include the actually-delivered counts in ITEMS_DONE so the dashboard can flag partial deliveries.

- [BUG] **`position` in WAREHOUSE_QUEUED is 1-indexed and correct at enqueue time, but it is a stale snapshot that is never re-sent as the queue drains (Check 9).** `position = #queue` (lines 224, 251) is sent once, at the moment of enqueue, and equals the entry's 1-based slot (the entry was just `table.insert`ed to the end, so `#queue` is its index). `position = 0` (lines 206, 301) correctly means "now being served." **The bug**: as earlier entries are served and `table.remove(queue, 1)`'d (line 298), every still-waiting entry's real position decreases, but the warehouse **never re-emits an updated WAREHOUSE_QUEUED** to those waiting turtles. A turtle queued at position 3 is told "3" once and keeps displaying 3 even after two ahead of it finish and it's actually next. So position is correct *only* at the instant of queuing; it goes stale immediately and is never refreshed (except on a duplicate ITEM_REQUEST, which re-sends the *current* `#queue`-based position only if it re-runs the enqueue path ‚Äî but a duplicate for an already-queued job hits the early-return at line 212 and re-sends nothing). Recommend broadcasting updated positions to all waiting entries whenever the queue head advances.

- [OVERSIGHT] **Logging covers state *entry* and aborts well, but there is no periodic "stuck in state X for Ys" heartbeat, so a silent stall is only diagnosable by inferring from the last log line (Check 10).** Positives: every state transition logs (Serving, "Turtle arrived", "CHESTS_READY sent ‚Äî waiting for CHESTS_PLACED", "Chests placed ‚Äî N batches", "Batch i/N exported ‚Äî waiting for BATCH_DONE", "All batches done ‚Äî waiting for ITEM_COLLECTED", "Job complete"), and every abort logs its reason with the jobId. So an operator reading the log can see which `wait`-state the warehouse last entered. **Gaps**: (a) once a state is entered, the warehouse goes **completely silent** until either the awaited message or the timeout (up to 120s, or 1800s in WAIT_ARRIVE) ‚Äî during a stall there is no "still waiting for CHESTS_PLACED (45s elapsed)" tick, so a live operator watching can't distinguish "hung" from "working." (b) There is no log of `state`/`stateTs`/`#queue`/`current.jobId` on a timer, and no `STATUS`/heartbeat sent to the server, so the dashboard has zero visibility into warehouse state (the warehouse is not in the registry). (c) The `tick()` IDLE branch returns silently when `#queue==0` (line 297), which is fine, but there's no startup/periodic "IDLE, queue empty" so a wedged-in-IDLE-with-a-non-empty-queue bug (shouldn't happen, but) would be invisible. Recommend a low-frequency (e.g. every 30s) state/queue-depth log line, and ideally a periodic heartbeat to the server for dashboard visibility.

- [OPT] **`abortCurrent` does not notify the turtle or server (cross-cutting, Checks 3/7/8).** All four abort sites (no chests in RS line 315, arrival timeout 323, CHESTS_PLACED timeout 359, BATCH_DONE timeout 393) call `abortCurrent`, which sweeps the EC and resets to IDLE but emits **no JOB_ABORT** to the turtle/server. The turtle, if alive, only discovers the abort via its own timeout. Sending JOB_ABORT (as the server does in fwdToTurtle, and as the warehouse already does for unknown mid-job messages at line 279) would let the counterparty fail fast instead of waiting out its own deadline. Low risk because both sides have timeouts, but it lengthens recovery.

- [OK] **Mid-job messages for unknown jobs are correctly aborted back to the turtle (Check, recovery path).** Lines 263-282: CHESTS_PLACED/BATCH_DONE/ITEM_COLLECTED carrying a jobId not in `current` or `queue` trigger a JOB_ABORT reply to `msg.from` (line 279), so a turtle stuck re-sending after a warehouse reboot is told to clean up rather than looping forever. This is the warehouse-side mirror of the server's NOT-FOUND handling and is the correct recovery for the reboot-amnesia case. (It also means a CHESTS_PLACED arriving slightly *before* the entry is popped from queue could race ‚Äî but the entry is checked against both `current` and `queue`, so an in-queue entry is recognized.)

## Layer 3: Dashboard / Bridge

### server.js

- [OK] **`pendingCommands` splice race is not a real race (Check 1).** Node is single-threaded with a run-to-completion event loop. The `/update` handler (line 145) is `async` but contains **no `await` before** `pendingCommands.splice(0)` (line 160) ‚Äî the body runs synchronously from `const { turtles, jobs, version } = req.body` down to `res.json({...commands: pendingCommands.splice(0)})` without yielding. (The `upsertMarker(...).catch(()=>{})` at line 152 is fire-and-forget ‚Äî it is **not** awaited, so it does not yield the handler.) Therefore two near-simultaneous `/update` POSTs are fully serialized: the first drains the whole array atomically and the second sees an empty array. No command is split across two responses, lost, or double-drained on the server side. **Caveat (real risk, but downstream):** commands are removed from `pendingCommands` the instant they are put in the response body, *before* the CC server has confirmed receipt. If the HTTP response is lost in transit (network drop, CC server crashes after sending the POST but before reading the body), those commands are gone forever ‚Äî there is no ACK/redelivery. This is an [OVERSIGHT] at the protocol level (at-most-once delivery for commands like RECALL_ALL/CLEAR_JOBS), not a JS concurrency bug.

- [PERF/OVERSIGHT] **`state.turtles` grows unbounded; offline turtles are never pruned (Check 2).** `state.turtles[id]` is only ever written (line 151, `{ ...state.turtles[id], ...data }`) ‚Äî there is no deletion path anywhere in `server.js`. This mirrors the `central_server.lua` registry finding: the bridge faithfully relays whatever the CC server pushes and never removes ids. Because the CC server itself never deletes registry entries (it only flips `online=false`), every turtle that has ever registered is pushed here and kept forever. Dashboard impact (confirmed in `public/index.html`): `renderFleet` (line 712) and `refreshTurtleSel` (line 777) iterate ALL of `turtles`, so a permanently-dead turtle stays in the fleet list as a greyed-out "OFFLINE" card (line 745) and in the turtle-select dropdown indefinitely. The map marker is only removed when the id vanishes from `turtles` (index.html line 626) ‚Äî which never happens ‚Äî so the stale Leaflet circle also persists (positioned at its last-known coords). The `online` flag the dashboard filters on (lines 695-696) is set entirely by the CC server; `server.js` does not compute liveness from `updatedAt`. Bounded in practice for a fixed fleet with stable `os.getComputerID` ids (re-registration overwrites the slot), but there is no decommission path and ghosts accumulate across the fleet's lifetime of churn.

- [OPT] **`markerExists` grows unbounded but is harmless (Check 3).** `markerExists[id]` (line 43) is set `true`/`false` per turtle id and never deleted. Same id-churn bound as `state.turtles` ‚Äî at most one entry per distinct turtle id ever seen. It is a small boolean map keyed by id, so memory impact is negligible; functionally it just tracks add-vs-update for the Dynmap marker. No cleanup needed. (Minor coupling: it is never reset to remove a marker when a turtle is decommissioned, but since there is no decommission path, `removeMarker` (line 99) is never actually called from anywhere in this file ‚Äî see PROTOCOL note below.)

- [PERF] **`rcon()` opens and tears down a fresh RCON connection on EVERY marker write (Check 4).** `rcon(cmd)` (lines 47-55) does `Rcon.connect()` ‚Üí `send` ‚Üí `client.end()` for a single command. In `/update`, `upsertMarker` is called once per turtle (line 152), and an *update* of an existing marker issues one `rcon` call (line 82) = one full TCP connect/auth/close. With N turtles pushed every ~2s, that is **N RCON connect+auth+close cycles every 2s** (and 2-3√ó N on the create/recreate path, lines 85-88, which fires up to three `rcon` calls for one marker). For a 10-turtle fleet that is ~5 connections/sec sustained, ~15+/sec on first push when markers are being created. Each connection re-runs the RCON auth handshake against the MC server. Risks: (a) RCON connection/auth overhead dominates; (b) some servers rate-limit or cap concurrent/total RCON logins and will start refusing connections under this churn (auth failures ‚Üí markers silently stop updating via the swallowed catch, see Check 9); (c) connect/end is async and these calls are fired un-awaited and in parallel across all turtles in the for-loop, so a burst of simultaneous connects hits the server at once. A single persistent/pooled RCON client reused across calls (connect once at startup, reconnect on error) would eliminate the churn.

- [PERF/OVERSIGHT] **`proxyDynmap` has no timeout on the upstream `http.get` (Check 5).** `proxyDynmap` (lines 111-118) and the `/dynmap-frame` handler (lines 135-140) call `http.get('http://127.0.0.1:8123...')` with no `timeout` option and never call `req.setTimeout(...)`. Node's default socket timeout for `http.get` is effectively none (no timeout) ‚Äî the `'error'` handler only fires on connection-level errors (ECONNREFUSED if Dynmap is down). If Dynmap is **up but slow/hung** (accepts the socket but never responds), the client browser request hangs indefinitely: the upstream socket stays open, `res` is never written, and the browser tab spins until the *browser's* own timeout. Worse, every such hung tile/asset request holds an open upstream socket and an open client response; under a Dynmap stall the dashboard can pile up many stuck connections. If Dynmap is fully **down**, `'error'` fires immediately ‚Üí `res.status(404).end()` (proxy) / `502` (frame), which is fine. Recommend `http.get(url, { timeout: 5000 }, ...)` plus an `'timeout'` handler that calls `req.destroy()` and `res.status(504).end()`.

- [BUG] **`state.version` is undefined until the first CC push (Check 6).** The initial `state` object (lines 36-40) declares `turtles`, `jobs`, `updatedAt` but **not** `version`. It is only created when a push includes one: `if (version) state.version = version` (line 157). So before the first `/update` (or if CC never sends `version`), `GET /state` (line 165) returns an object with `version` absent (`undefined`, omitted from JSON). Dashboard handling (index.html line 675): `if (d.version) document.getElementById('cc-version').textContent = \`v${d.version}\`` ‚Äî the guard means the version span keeps its placeholder `‚Äî` (index.html line 232) rather than showing a wrong value, so the header just shows a dash until the first push lands. Not a crash; cosmetic. Initialize `version: null` in the `state` literal for clarity/consistency with the other fields.

- [OVERSIGHT] **Default `express.json()` 100kb body limit could reject a large fleet push (Check 7).** `app.use(express.json())` (line 16) uses the default `limit: '100kb'`. The `/update` body is `{ turtles, jobs, version }` where `turtles` is the full registry (every turtle: x/y/z/status/fuel/role/jobId/dock/online ‚âà a few hundred bytes serialized each) and `jobs` is the full job array. For a small fleet this is a few KB. But it scales linearly: a large fleet (dozens of turtles) plus a long jobs queue, each job carrying `params.destination` and metadata, could approach or exceed 100kb. If it does, Express responds **413 Payload Too Large** and the route never runs ‚Äî the CC server's push silently fails (it likely just logs "Bridge push failed" per the central_server audit) and the dashboard goes stale with no on-screen indication. Set an explicit generous limit, e.g. `express.json({ limit: '5mb' })`, to remove the cliff. Currently latent for the present fleet size but a scaling landmine.

- [OVERSIGHT] **No authentication on `/command` (or any route) ‚Äî intentional but unguarded (Check 8).** `POST /command` (lines 169-175) accepts any `{ type, params }` and queues it for the CC server to execute, with no auth token, origin check, or rate limit. Anyone who can reach port 3000 can queue `RECALL_ALL`, `CLEAR_JOBS`, `DISPATCH_DELIVERY`, `UPDATE_ALL` (OTA reboot), etc. `/update` is equally open ‚Äî anyone can POST fake turtle state and poison the registry/markers, and can also **drain** `pendingCommands` (the `/update` response returns and clears them), so an attacker polling `/update` could steal/cancel legitimately-queued dashboard commands. Given this is a LAN-only hobby dashboard with a hardcoded RCON password in source (line 26), the lack of auth is presumably intentional. Flagging as [OVERSIGHT]: if the box is ever exposed beyond the trusted LAN, this is a full unauthenticated control plane for the turtle fleet and the Minecraft server (via queued commands). At minimum bind to localhost or a private interface and/or add a shared-secret header.

- [BUG/OVERSIGHT] **`upsertMarker` swallows errors and can wedge `markerExists[id]` into a permanent recreate loop (Check 9).** In `/update`, marker calls are fired as `upsertMarker(...).catch(()=>{})` (line 152) ‚Äî the outer rejection is silenced. Inside `upsertMarker`, the `try` already catches RCON failures (line 93) and **sets `markerExists[id] = false`** (line 95) on any error, logging to console. Consequences: (a) if RCON is down, every marker write throws, `markerExists[id]` is forced back to `false`, and the next push takes the **create** branch (line 83) instead of update ‚Äî so once RCON recovers, the marker will be re-created (or hit the delete+add recreate path, lines 87-88), which is actually self-healing. (b) The genuinely bad state: an error during the *create* branch sets `markerExists[id]=false` (line 95) even though a partial `dmarker add` may have *succeeded* on the server ‚Äî the next push re-enters create, the plain `add` throws "marker already exists" (caught at line 86), and the fallback does `delete` then `add` (lines 87-88). That recreate path works but means a flapping RCON connection causes the marker to be deleted and re-added on every single push (visual flicker on the map, extra RCON load ‚Äî compounding Check 4). (c) Errors are only visible in the server console (`console.error` line 94); the dashboard has no signal that markers stopped updating ‚Äî they just freeze at last-known position. The double-swallow (inner catch sets state + logs, outer `.catch(()=>{})` discards the resolved-undefined) means a transient RCON blip silently degrades the map with no surfaced error.

- [PROTOCOL] **`removeMarker` is dead code ‚Äî defined, never called (Check 3/9-adjacent).** `removeMarker(id)` (lines 99-107) is the only path that would delete a Dynmap marker and reset `markerExists[id]=false` cleanly, but nothing in `server.js` ever calls it. Combined with the no-prune findings (Checks 2/3), a turtle's marker, once created, is never intentionally removed ‚Äî it lingers on the map at its last position for the life of the process. Either wire `removeMarker` into an offline/decommission path or remove it.

- [OVERSIGHT] **`/update` partial-payload handling is permissive and mostly safe, but a `version`-only push is rejected (Check 10).** The guard is `if (!turtles && !jobs) return 400` (line 147). So: (a) `turtles` without `jobs` ‚Üí fine, jobs untouched (the `if (jobs)` at line 156 is skipped, `state.jobs` keeps its prior value). (b) `jobs` without `turtles` ‚Üí fine, the turtle loop (line 149) is skipped. (c) `version` missing ‚Üí fine, `state.version` simply isn't updated (line 157 guarded). (d) **Edge case:** a push of `{ version: "1.3.37" }` with neither `turtles` nor `jobs` is **rejected 400** even though it carries valid data ‚Äî `version` alone can never update `state.version` because the guard fails first. Unlikely in practice (CC always sends turtles+jobs), but the guard should arguably also accept a version-only heartbeat, or at least the guard logic (`!turtles && !jobs`) should be documented as "must carry at least one of turtles/jobs." Also note `if (turtles)` / `if (jobs)` are truthiness checks: an explicitly-empty `turtles: {}` is **falsy-skipped? no** ‚Äî `{}` is truthy, so an empty object enters the for-loop harmlessly (zero iterations); but an empty `jobs: []` is also truthy and correctly sets `state.jobs = []` (a real "all jobs cleared" push). Behavior is correct for empty collections. No crash on any missing-field combination; only the version-only case is a (benign) false-400.
